# v2.2 Design — Encrypted Memory via git-crypt

**Status**: Draft
**Target**: bobusang v2.2
**Author**: @jun0-ds
**Date**: 2026-04-14
**Supersedes**: [v2.1-private-tags.md](./v2.1-private-tags.md) (killed by devil's advocate review)

## Problem

Users of bobusang will write sensitive content into memory files: API keys, internal names, pricing, contact info, half-baked thoughts they don't want leaking. The git remote — even if private — is not a trust boundary that can hold forever. Private repos get breached, visibility flips by accident, hosting companies get compromised, old collaborators retain access. A multi-device memory system needs to assume the remote is eventually hostile.

v2.1 tried to solve this with `<private>` tags and a git clean filter. [Devil's advocate review](./v2.1-private-tags.md) found the design was fail-OPEN on fresh clones, destroyed secrets on routine `git checkout --`, and broke multi-device sync (the whole point of bobusang). It was killed.

v2.2 picks the honest primitive: **encrypt the files. Let the encrypted form travel through git. Manage keys out-of-band.**

## Threat model (explicit)

What v2.2 defends against:

1. **Remote repo leak.** GitHub visibility flips, backup gets exposed, account compromised. Attacker sees encrypted blobs, not plaintext.
2. **Long-lived access drift.** A collaborator from two years ago still has read access to the remote. They can't decrypt what was committed after they left (assuming key rotation, which we'll document but not enforce).
3. **Local device loss without key.** Laptop stolen. If the key lives in a password manager / hardware token / separate sync path, the thief has the repo but not the key. Encrypted memory is useless to them.

What v2.2 does **not** defend against (stated upfront, not buried):

1. **Compromise of a device with the key loaded.** If the attacker gets a live shell on a machine that has the decrypted working tree, everything is readable. v2.2 is not a TPM, not a sandbox, not an HSM.
2. **Commit messages, file names, directory structure.** git-crypt encrypts file *contents*. Names and structure are visible. Users must not put secrets in commit messages or filenames.
3. **Git history from before v2.2.** Anything committed in plaintext before v2.2 is in history forever. `git filter-repo` is documented but not automated — rewriting history is destructive and must be the user's deliberate choice.
4. **Key compromise is retroactive.** git-crypt uses a static repository key. If the key leaks, *all* historical encrypted content is decryptable. Key rotation requires re-encrypting, which is a manual procedure with real cost.
5. **Compliance regimes.** Not HIPAA, not SOC2, not PCI. Use a real vault.
6. **Non-text secrets in images, screenshots, binary artifacts.** git-crypt encrypts them too, but bobusang doesn't specifically track them and deduplication/diff is poor on binaries.

## Why git-crypt (and not alternatives)

| Option | Fit | Reason |
|--------|-----|--------|
| **git-crypt** | ✅ chosen | File-level encryption at the git boundary. Author doesn't tag anything — `.gitattributes` decides which files are encrypted. Plaintext on disk for Claude to read. Ciphertext in git for safety. Actually syncs across devices (the encrypted form travels). Mature, widely used, audited enough. |
| git-secret | ❌ | Per-file GPG wrapping. Requires re-encrypt on every change to the recipient list. Clumsier multi-device story. |
| sops | ❌ | Optimized for structured config (YAML/JSON) with partial encryption. Wrong shape for free-form markdown memory. |
| age + manual scripts | ❌ | No git integration. User manages encrypt/decrypt cycles by hand. Error-prone in the exact way bobusang is supposed to eliminate. |
| `<private>` tags + clean filter | ❌ | Killed by devil (see v2.1 doc). |
| Gitignored per-device file | ⚠ partial | Solves nothing for multi-device. Documented as a fallback for users who refuse key management. |

git-crypt's limitations are real but **bounded and namable**. The v2.1 design's limitations were unbounded (every smudge op, every missing-filter clone, every typo). Bounded failure modes are engineerable. Unbounded ones aren't.

## Design

### What gets encrypted

`.gitattributes`:

```
memory/secrets/** filter=git-crypt diff=git-crypt
notes/secrets/**  filter=git-crypt diff=git-crypt
```

**Encryption is directory-scoped, not tag-scoped.** A file is either entirely encrypted or entirely plaintext, based on which directory it lives in. No regex parsing, no author discipline, no partial files.

```
memory/
├── core/              # plaintext — identity, work style (not secret)
├── domain/            # plaintext — active projects (non-sensitive parts)
├── secrets/           # ENCRYPTED — anything sensitive
│   ├── credentials.md
│   ├── internal-names.md
│   └── project-x-secrets.md
└── archive/           # plaintext — graduated projects
```

Users who want to mark an entire domain memory as sensitive put it under `secrets/`. Users who want to mix: split the file into `domain/project-x.md` (public) + `secrets/project-x.md` (private), and let Claude read both via a CLAUDE.md rule.

### CLAUDE.md integration

New section in the default `CLAUDE.md` template:

```markdown
## Secrets

Files under `~/.claude/memory/secrets/` and `~/.claude/notes/secrets/` are
encrypted at rest in git via git-crypt. On a device where the key is loaded,
they appear as normal markdown and should be read alongside their public
counterparts when relevant.

When you see a `memory/domain/foo.md`, also check for `memory/secrets/foo.md`.
Treat the secrets file as the private appendix to the public one.
```

This is a convention, not a mechanism. bobusang doesn't enforce the filename pairing. If users want a different layout they can adjust the rule.

### Key lifecycle

This is the part git-crypt doesn't solve for you, and the part where v2.2 earns its keep.

#### Initial setup (device A — the originating device)

`setup.sh` gains an interactive path:

```
$ bash setup.sh
...
Do you want to enable encrypted memory (via git-crypt)?
  [y] Yes — I will manage a key across my devices
  [n] No  — all memory will be plaintext in the remote

> y

Installing git-crypt... ok
Initializing git-crypt in this repo... ok
Exporting key to: ~/.claude/bobusang-key.gpg
  ⚠ This file IS the decryption key for your encrypted memory.
  ⚠ Back it up somewhere safe (password manager, encrypted USB, hardware token).
  ⚠ Do NOT commit it. Do NOT put it in the same git remote.
  ⚠ If you lose this file and have no backup, encrypted memory is UNRECOVERABLE.

To add another device:
  1. Copy ~/.claude/bobusang-key.gpg to the new device via a trusted channel
     (USB, Signal, encrypted sync, NOT the same git remote).
  2. On the new device: git clone <remote> && cd <repo> && git-crypt unlock ~/path/to/bobusang-key.gpg
  3. Run bash setup.sh on the new device as usual.

Enable `memory/secrets/` and `notes/secrets/` as encrypted paths? [Y/n] y
Done. Your `memory/secrets/` directory is now encrypted on commit.
```

#### Adding a new device (device B)

Documented, not automated. The key has to travel through a trusted side-channel — bobusang cannot and should not automate key distribution. Options documented in `docs/guides/multi-device-keys.md`:

1. **pCloud/Dropbox with its own encryption.** Store the key file in a sync folder that has separate credentials from the git remote. Two-factor-ish: attacker needs both.
2. **Password manager with file attachment.** 1Password, Bitwarden, KeePassXC all support binary attachments. Preferred for most users.
3. **Hardware security key with file storage.** YubiKey, SoloKey. Overkill for most.
4. **Physical transfer.** USB stick, printed QR code of the base64-encoded key. The "I don't trust the cloud" option.

**Explicit non-recommendation**: do not put the key in a note on the same device you're trying to protect. Do not email it to yourself. Do not commit it anywhere, even in a "just temporarily" way.

#### Key loss

Documented in the same guide. Key loss = encrypted memory is gone. No recovery path. This is the cost of real encryption and should be stated plainly, not softened. Mitigation: always have at least two backup copies of the key in two different trust domains.

#### Key rotation

Not automated in v2.2. Documented procedure:

1. Decrypt all encrypted files to plaintext temporarily
2. Remove git-crypt state (`git-crypt lock` + manual cleanup)
3. Re-init git-crypt with new key
4. Re-encrypt
5. Force-push (destructive — will break other devices until they get the new key)

This is deliberately painful because key rotation should be a deliberate decision, not a routine.

### What bobusang actually builds

git-crypt is the engine. bobusang ships:

1. **`setup.sh` integration**: detect git-crypt, install if missing (via package manager, with explicit confirmation), init repo, export key, print key-handling warnings.
2. **`.gitattributes` template**: pre-wired with the `memory/secrets/**` and `notes/secrets/**` patterns. User can edit if they want different paths.
3. **`CLAUDE.md` template update**: the Secrets section above.
4. **`docs/guides/multi-device-keys.md`**: honest, long-form guide on key handling. Covers the four distribution options, loss scenarios, rotation procedure, what to do if a device is compromised.
5. **`sync-memory.sh` awareness**: detect when the repo uses git-crypt but the current shell has no key loaded (working tree shows encrypted blobs). Print a clear error — "this repo uses git-crypt and no key is loaded; run `git-crypt unlock <keyfile>` and re-run setup" — instead of silently syncing garbage.
6. **`detect-changes.sh` awareness**: same treatment. Don't falsely flag "all files changed" when the real problem is "key not loaded."
7. **Pre-commit sanity check** (optional, opt-in): a hook that refuses to commit if `memory/secrets/**` files appear to contain plaintext markdown (heuristic: starts with `---\n` frontmatter and is human-readable). Catches the "user forgot to run `git-crypt unlock` and is now committing decrypted files that will re-encrypt on next commit — but also the case where encryption isn't active at all because setup was skipped."

### Answers to the devil findings (v2.1 → v2.2)

| v2.1 flaw | v2.2 answer |
|-----------|-------------|
| Fail-OPEN on fresh clone (`.git/config` not committed) | git-crypt state *is* committed. Fresh clone sees encrypted blobs. No key = no plaintext, period. Fail-CLOSED by construction. |
| Multi-device sync broken (smudge = cat) | Encrypted files sync normally. Device B with the key gets plaintext; without the key, gets blobs. Sync works for everyone with the key. |
| `git checkout --` destroys secrets | Working tree and index are both encrypted-or-decrypted consistently. No asymmetry. Checkout is safe. |
| Opt-in contradicts fail-closed | Opt-in selects whether encryption is active *at all*. Once enabled, it's enforced at the file level, not the tag level. Users can't accidentally skip encryption on an individual file under `secrets/`. |
| Author-controlled typos leak | No tags to typo. Directory membership is the only control. |
| Commit messages leak | Still true. Documented in threat model as user's responsibility. No technical solution exists. |
| History never scrubbed | Still true for pre-v2.2 commits. Documented. `git filter-repo` guide linked, not automated. |
| Wrong abstraction from claude-mem | claude-mem's `<private>` is for a daemon-internal strip pipeline. v2.2 uses directory-scoped encryption at the git boundary, which is the correct layer for bobusang's architecture. Different mechanism, different layer, right tool for the right job. |

### New failure modes v2.2 introduces (honest accounting)

1. **Key distribution is now a user problem.** bobusang cannot help beyond documentation. Users who mishandle the key (e.g., sync it via the same service as the repo) lose the protection. Mitigation: strong, repeated warnings in setup output and docs.
2. **Key loss is terminal.** No recovery. Documented upfront, not buried.
3. **git-crypt tooling dependency.** Installing it is a package manager operation and can fail on some systems (Alpine, Termux, locked-down work machines). `setup.sh` must handle "git-crypt not installable here" gracefully and let the user fall back to plaintext memory with an explicit acknowledgment.
4. **Mixed-encryption confusion.** A user might forget whether a given piece of memory went into `domain/` or `secrets/`. Search won't look in `secrets/` without the key loaded. Documented as "when grepping memory, be aware that `secrets/` is only searchable when unlocked."
5. **Diff noise on binary view.** `git diff` on encrypted files without the key shows binary-ish output. Users need to know this is normal.
6. **Merge conflicts on encrypted files.** git-crypt handles them but the conflict markers appear in the plaintext version, which on re-encrypt can leak partial content if not cleaned carefully. Documented.

These are real costs. They are also finite, documentable, and mostly one-time-learning. The v2.1 costs were unbounded.

## Scope

**In scope for v2.2:**
- `setup.sh` git-crypt installation + init + key export
- `.gitattributes` template with `secrets/**` patterns
- `CLAUDE.md` template update
- `docs/guides/multi-device-keys.md` (long-form)
- `sync-memory.sh` and `detect-changes.sh` key-loaded awareness
- Optional pre-commit sanity check

**Out of scope for v2.2 (future work):**
- Automated key distribution (requires real infrastructure — maybe never)
- Key rotation automation
- Plaintext→encrypted migration tool for existing bobusang users
- GUI for key management

**Explicit non-goals:**
- Becoming a secrets manager
- Protecting against device compromise
- Compliance certifications
- Binary asset encryption strategy

## Open questions

1. **Default encrypted paths**: only `secrets/` subdirs, or also encrypt `notes/{hostname}.md` by default (since per-device notes often contain fresh, sensitive working context)?
2. **Install strategy**: auto-install git-crypt via package manager, or just print installation instructions and let the user do it? Auto-install is friendlier but risks surprising `sudo` prompts on servers.
3. **Key file default location**: `~/.claude/bobusang-key.gpg` (inside the claude dir — discoverable, but near the repo it protects) vs `~/.config/bobusang/key.gpg` (XDG, further from the repo) vs prompting every time.
4. **Pre-commit sanity check**: ship it as on-by-default, opt-in, or not at all in v2.2? Lean opt-in — it's a nice-to-have and shouldn't block the core ship.
5. **Plaintext fallback UX**: when a user answers "no" to encryption, should bobusang add a `.gitattributes` comment reminding them later? Or stay silent to respect the decision?
6. **Handling `git pull` on an unlocked repo with new encrypted files**: does git-crypt smudge them correctly after pull? Need to test. If not, document the `git-crypt unlock` re-run requirement.

## Testing plan

- **Unit**: `setup.sh` paths for (a) git-crypt already installed, (b) not installed but installable, (c) not installable. Key export to expected location with correct permissions (0600).
- **Integration**: Full cycle on a throwaway repo — init, encrypt a file, commit, clone to "device B," unlock with exported key, verify plaintext, modify, commit, pull back to "device A," verify consistency.
- **Failure modes**: Clone without key → verify `sync-memory.sh` errors clearly. Lose key simulation → verify docs match reality. Pre-commit sanity check triggers correctly on plaintext-in-secrets.
- **Platform matrix**: WSL2 Ubuntu, native Linux, macOS. git-crypt installation path differs on each.

## Rollout

- **v2.2.0**: feature lands. `setup.sh` interactive prompt. Default is **no encryption** (preserves v2.0/v2.1-era behavior for existing users). New users who answer "yes" get the full path.
- **v2.2.1**: bug fixes based on real-world install experiences.
- **v2.3**: if key-distribution pain is severe, revisit — possibly with an age-based alternative for users who prefer a simpler keygen story. Not committed.
