# v2.2 Design — Encrypted Memory via git-crypt

**Status**: Draft (rev 2 — post-devil revisions)
**Target**: bobusang v2.2
**Author**: @jun0-ds
**Date**: 2026-04-14 (rev 1), 2026-04-14 (rev 2)
**Supersedes**: [v2.1-private-tags.md](./v2.1-private-tags.md) (killed by devil's advocate review)
**Revision history**:
- rev 1 — initial git-crypt design after v2.1 was killed
- rev 2 — devil's advocate review on v2.2 itself surfaced 8 weaknesses. This revision addresses them: auto-unlock reality in threat model, password-manager-only key distribution story, file-move silent re-encryption trap, git-crypt ecosystem audit as pre-implementation blocker, mandatory pre-commit sanity check, git pull + smudge as implementation spike, structural key-loss defense, platform matrix gating.

## Problem

Users of bobusang will write sensitive content into memory files: API keys, internal names, pricing, contact info, half-baked thoughts they don't want leaking. The git remote — even if private — is not a trust boundary that can hold forever. Private repos get breached, visibility flips by accident, hosting companies get compromised, old collaborators retain access. A multi-device memory system needs to assume the remote is eventually hostile.

v2.1 tried to solve this with `<private>` tags and a git clean filter. [Devil's advocate review](./v2.1-private-tags.md) found the design was fail-OPEN on fresh clones, destroyed secrets on routine `git checkout --`, and broke multi-device sync (the whole point of bobusang). It was killed.

v2.2 picks the honest primitive: **encrypt the files. Let the encrypted form travel through git. Manage keys out-of-band.**

A second devil's advocate round on v2.2 itself confirmed the architecture is sound (git-crypt is the right primitive at the right layer) but surfaced weaknesses in how the design presented its own limits. This revision bakes those limits into the main text instead of softening them in caveats.

## Threat model (explicit)

### What v2.2 defends against

1. **Remote repo leak, device not compromised.** GitHub visibility flips, backup gets exposed, account compromised via token leak. Attacker sees encrypted blobs, not plaintext. This is the primary defended case.
2. **Long-lived access drift (theoretical).** A collaborator from two years ago still has read access to the remote. They can't decrypt content committed after their key access was revoked — **but only if the key was actually rotated**, which v2.2 treats as essentially not-happening in practice (see key rotation section). So in practice this defense is "access drift during the period before the collaborator ever had the key," which is narrower.
3. **Laptop theft with key stored separately.** If the decryption key is in a password manager or on a physical token and NOT cached on the stolen laptop, the thief has the repo but not the key.

### What v2.2 does NOT defend against (stated upfront, not buried)

1. **Compromise of a device with the key loaded.** If the attacker gets a live shell on an unlocked device, everything is readable.
2. **Auto-unlock convenience configurations.** *(new in rev 2)* The ergonomics of git-crypt push users toward caching the key on disk (`~/.claude/bobusang-key.gpg`) or in a GPG agent with indefinite cache, so that `git-crypt unlock` doesn't require a passphrase every session. **On any device configured this way, the "key is separate" assumption collapses.** Device compromise = key compromise. v2.2 does not prevent users from configuring auto-unlock, because the alternative (passphrase every session) is untenable for daily use. Realistically, most devices will end up auto-unlocked. **This makes v2.2 a one-factor defense against a remote-only leak, not a two-factor defense against arbitrary compromise.** Calling it anything else is dishonest.
3. **Commit messages, file names, directory structure.** git-crypt encrypts file *contents*. Names and structure are visible. Users must not put secrets in commit messages or filenames. No technical mitigation — documented as author responsibility and accepted as a leak vector.
4. **Git history from before v2.2.** Anything committed in plaintext before v2.2 is in history forever. `git filter-repo` is documented but not automated — rewriting history is destructive and must be the user's deliberate choice.
5. **Key compromise is retroactive and effectively unrecoverable.** git-crypt uses a static repository key. If the key leaks, *all* historical encrypted content is decryptable, and in practice users with multi-month memory across 4 devices **will not rotate**. The honest recovery path is "accept the breach and consider whether to start a new repo from scratch." Rotation is documented but should not be presented as a realistic ongoing mitigation.
6. **Compliance regimes.** Not HIPAA, not SOC2, not PCI. Use a real vault.
7. **Non-text secrets in images, screenshots, binary artifacts.** git-crypt encrypts them too, but bobusang doesn't specifically track them and deduplication/diff is poor on binaries.
8. **File-move silent re-encryption.** *(new in rev 2)* See "New failure modes" below. Moving a file between encrypted and plaintext directories silently changes its encryption status with no error and no user warning. Defended by a mandatory pre-commit check (see below), not by git-crypt itself.

### Summary of the honest story

v2.2 is **one-factor encryption at rest, primarily protecting against remote-only exposure**. It is not a vault, not a two-factor scheme, and not a protection against device compromise. It is a large improvement over "plaintext in a private repo that might someday leak," and that is the specific threat it targets.

## Why git-crypt (and pre-implementation ecosystem audit)

### Comparison

| Option | Fit | Reason |
|--------|-----|--------|
| **git-crypt** | ✅ chosen (pending audit) | File-level encryption at the git boundary. `.gitattributes` decides encryption scope — no inline tags, no author discipline. Plaintext on disk for Claude to read. Ciphertext in git. Actually syncs across devices. Widely used in infrastructure contexts. |
| git-secret | ❌ | Per-file GPG wrapping. Requires re-encrypt on every recipient-list change. Clumsier multi-device story. |
| sops | ❌ | Optimized for structured config (YAML/JSON) with partial encryption. Wrong shape for free-form markdown memory. |
| age + manual scripts | ❌ | No git integration. User manages encrypt/decrypt cycles by hand. Error-prone in exactly the way bobusang is meant to eliminate. |
| **git-agecrypt** | ⚠ evaluate in audit | Age-based alternative to git-crypt. More modern primitives. Actively maintained. Must be evaluated as a fallback if git-crypt ecosystem audit fails. |
| **transcrypt** | ⚠ evaluate in audit | Shell-script wrapper around openssl. Simpler, older, different trade-offs. Also evaluate in audit. |
| `<private>` tags + clean filter | ❌ | Killed by v2.1 devil review. |
| Gitignored per-device file | ⚠ fallback | Solves nothing for multi-device. Documented as a fallback for users who refuse key management entirely. |

### Ecosystem audit (BLOCKING — must complete before implementation)

git-crypt is maintained by a single person (Andrew Ayer). The last release (0.7.0) was in early 2022. Three-plus years without a release for a security-critical tool is a yellow flag, not a red one — it may be stable rather than abandoned — but the design is not confirmed until an audit answers:

1. **Activity**: Last commit on `master`, issue response cadence in the past 12 months, any public statement from the maintainer about project status.
2. **Security**: Known CVEs or unfixed security issues. Recent dependency updates. Any forks that address the original's gaps.
3. **Compatibility**: Works with current git releases. Handles the file operations bobusang performs (pull, merge, rename, checkout, stash) without corruption.
4. **Alternatives comparison**: Hands-on evaluation of git-agecrypt and transcrypt for the specific bobusang use case. Which has fewer footguns for a non-expert user?

Output of the audit is a short decision memo committed to `docs/design/ecosystem-audit-2026-04.md` with one of three verdicts:

- **git-crypt confirmed** — implement v2.2 as specified
- **switch to git-agecrypt (or other)** — rev 3 of this design doc, swapping the primitive
- **no primitive is ready** — v2.2 is paused; revisit when the ecosystem matures; ship plaintext-only until then

**No implementation work begins until this audit is committed.**

## Design

### What gets encrypted

`.gitattributes`:

```
memory/secrets/** filter=git-crypt diff=git-crypt
notes/secrets/**  filter=git-crypt diff=git-crypt
```

**Encryption is directory-scoped, not tag-scoped.** A file is either entirely encrypted or entirely plaintext, based on which directory it lives in. No regex parsing, no author discipline on a per-line basis.

```
memory/
├── core/              # plaintext — identity, work style
├── domain/            # plaintext — active projects (non-sensitive parts)
├── secrets/           # ENCRYPTED — anything sensitive
│   ├── credentials.md
│   ├── internal-names.md
│   └── project-x-secrets.md
└── archive/           # plaintext — graduated projects
```

Users who want to mix: split a file into `domain/project-x.md` (public) + `secrets/project-x.md` (private), and let Claude read both via a CLAUDE.md rule.

### The file-move trap (addressed, not ignored)

Directory-scoped encryption has a failure mode v2.1's tag-scoped approach did not: **moving a file between encrypted and plaintext directories silently changes its encryption status with no feedback.**

```bash
# User intent: "re-categorize this as public info"
mv memory/secrets/project-x.md memory/domain/project-x.md
git add memory/
git commit -m "move project-x to public section"
git push
# Result: entire file contents now plaintext on the remote
```

And the reverse:

```bash
mv memory/domain/foo.md memory/secrets/foo.md
# Result: future commits encrypted, but ALL prior plaintext history of foo.md
# remains in git history forever. User gets a false sense of "I encrypted it."
```

This is directly analogous to v2.1's "`git checkout --` destroys secrets" bug — a routine git operation silently changes encryption state without warning. v2.2 must not ship without a defense.

**Defense: mandatory pre-commit check.** A hook bobusang installs at `setup.sh` time. On every commit, it runs `git-crypt status` on all staged files and compares the expected encryption state (based on current `.gitattributes`) against the actual staged blob (encrypted or plaintext). If any staged file under `secrets/**` is plaintext, or any file outside `secrets/**` that was previously encrypted is now plaintext (move detection), the commit is refused with a clear message:

```
bobusang: refusing commit — encryption state mismatch

  memory/domain/project-x.md appears to be plaintext but its prior
  revision at memory/secrets/project-x.md was encrypted.

  If you intend to publish this content in plaintext, run:
      bobusang reclassify memory/secrets/project-x.md --to-plaintext
  This will explicitly acknowledge the encryption change.

  If this was a mistake, move the file back to secrets/.
```

This check is **mandatory**, not opt-in. Without it, the file-move trap is unmitigated, and the design has the same "routine git op silently compromises secrets" flaw that killed v2.1.

The `bobusang reclassify` command is a new addition: the only supported way to change a file's encryption status. It decrypts (if needed), moves, optionally clears history if the user explicitly asks and accepts the destructive rewrite, and re-commits with a recognizable commit message so the pre-commit check knows to allow it.

### CLAUDE.md integration

New section in the default `CLAUDE.md` template:

```markdown
## Secrets

Files under `~/.claude/memory/secrets/` and `~/.claude/notes/secrets/` are
encrypted at rest in git via git-crypt. On a device where the key is loaded,
they appear as normal markdown and should be read alongside their public
counterparts when relevant.

When you see `memory/domain/foo.md`, also check for `memory/secrets/foo.md`.
Treat the secrets file as the private appendix to the public one.

Never move files between `secrets/` and non-secrets directories with `mv` or
`git mv`. Use `bobusang reclassify` so the encryption state change is
explicit and recorded.
```

## Key lifecycle

### Key distribution: one practical answer, not four

Earlier drafts listed four options for getting the decryption key onto new devices: cloud with separate credentials, password manager, hardware token, physical transfer. In reality, **only the password manager is practical for a daily-use multi-device workflow.**

- **Cloud with separate credentials**: requires the user to maintain a second cloud account with genuinely independent credentials (different email, different password, different 2FA). Most users recycle credentials. If the "separate" cloud shares any auth surface with GitHub, the separation is fiction.
- **Hardware token**: great in theory. Unrealistic to carry across 4 devices (laptop, sub-laptop, GPU dev server, GPU prod server) for daily sessions. Falls back to "left plugged into one device," which degenerates to "key is on that device's disk."
- **Physical transfer (USB, printed QR)**: impossible for remote servers (SSH-only access, no physical presence). Degenerates to SCP, which shares the SSH trust surface with everything else on that device.
- **Password manager with file attachment**: 1Password, Bitwarden, KeePassXC all support binary attachments. Credentials live in the user's existing password manager (already presumably well-protected). Works on all platforms. Practical for daily use. **This is the supported path.**

v2.2 documentation and `setup.sh` prompts should **recommend password manager as the default and only first-class option**, and treat the others as acknowledged edge cases with clear warnings. Pretending there's a flexible menu is how users end up in insecure configurations.

### Structural defense against key loss

A warning message ("⚠ if you lose this key, memory is unrecoverable") is not a defense. Users read and dismiss. The design enforces key backup at structural points:

1. **setup.sh requires two backup destinations before it will complete encryption setup.**
   ```
   You are enabling encrypted memory. bobusang will not proceed until you have
   backed up the decryption key in two separate locations.

   Location 1 (e.g., your password manager): [prompt]
   Location 2 (e.g., encrypted USB, second password manager): [prompt]

   bobusang will not verify these locations — it trusts you to have put the
   key there. But it will remind you to verify monthly.

   To bypass this check (NOT recommended): --i-accept-single-point-of-failure
   ```
2. **`bobusang verify-key-backup` command.** User runs it. bobusang prompts them to manually retrieve the key from each of their claimed backup locations and confirm its SHA256 matches. On success, records the timestamp.
3. **Periodic reminder in session-start hook.** If `verify-key-backup` has not been run in 60 days, the session-start output shows:
   ```
   bobusang: key backup last verified 73 days ago.
   Run `bobusang verify-key-backup` to confirm you can still recover your memory.
   ```
   The user can dismiss this, but the cost of dismissing is seeing the reminder again next session. Nag is the point.
4. **No automatic recovery channel.** bobusang does not ship a "cloud backup of the key" feature, even opt-in. The moment such a feature exists, it becomes a target and a crutch. Users who want belt-and-braces can use their existing backup tooling on the key file.

### Initial setup (device A)

```
$ bash setup.sh
...
Enable encrypted memory (via git-crypt)? [y/N] y

Running ecosystem audit check... found docs/design/ecosystem-audit-2026-04.md
Verdict: git-crypt confirmed. Proceeding.

Installing git-crypt...
  [detects package manager, prompts for sudo if needed, or aborts with
   "install git-crypt manually and re-run" if no installable path]

Initializing git-crypt in this repo...
Exporting key to: ~/.claude/bobusang-key.gpg (mode 0600)

⚠ This file IS the decryption key for all your encrypted memory.
⚠ Back it up in TWO separate locations before proceeding.

Location 1 (e.g., password manager): [prompt]
Location 2 (e.g., encrypted USB): [prompt]

Installing pre-commit check... ok
Installing session-start key-backup reminder... ok

Writing .gitattributes with:
  memory/secrets/** filter=git-crypt diff=git-crypt
  notes/secrets/**  filter=git-crypt diff=git-crypt

Done. See docs/guides/multi-device-keys.md for adding other devices.
```

### Adding a new device (device B)

Documented in `docs/guides/multi-device-keys.md`. The procedure:

1. Retrieve the key from the password manager on device B.
2. `git clone <remote> && cd <repo>`
3. `git-crypt unlock /path/to/retrieved-key.gpg`
4. `bash setup.sh` — bobusang recognizes the unlock state and installs the pre-commit check and session-start hooks without re-exporting a new key.
5. Optionally run `bobusang verify-key-backup` to confirm device B's retrieval path still works.

### Key rotation (honest answer)

**Realistic answer: users will not rotate.** Key rotation in git-crypt requires decrypting everything, re-initializing the repo, re-encrypting, force-pushing, and coordinating all devices to re-unlock with the new key. With 4 devices and months of memory, this is effectively a migration project, not a routine operation.

The documented response to key compromise is therefore **not rotation**. It is:

1. Stop using the current repo for any new sensitive content immediately.
2. Treat the current repo's encrypted content as compromised going forward.
3. Decide whether to migrate to a fresh repo with a new key, accepting the loss of some history continuity, or to accept the compromise and continue with the existing key on the assumption that the leaked key has not actually reached a motivated attacker.

This is not glamorous, but it is what real users will do. Pretending rotation is a routine recovery path invites users to underinvest in protecting the current key.

## What bobusang actually builds

1. **Ecosystem audit** (`docs/design/ecosystem-audit-2026-04.md`) — blocking.
2. **`setup.sh` integration**: git-crypt detection, installation path or clear abort, init, key export, **two-backup enforcement**, pre-commit hook install, session-start reminder install.
3. **`.gitattributes` template** with `memory/secrets/**` and `notes/secrets/**` patterns.
4. **`CLAUDE.md` template update** with the Secrets section (including the "never `mv` across encryption boundaries" rule).
5. **`docs/guides/multi-device-keys.md`** — long-form honest guide. Password manager as the first-class path. Other options as edge cases. Key loss framed as terminal. Key rotation framed as migration-level, not routine.
6. **`sync-memory.sh` awareness**: detect git-crypt lock state and refuse to sync with a clear message if the key isn't loaded on a repo that uses encryption. Don't silently sync garbage.
7. **`detect-changes.sh` awareness**: same — don't falsely flag "all files changed" because the key isn't loaded.
8. **Pre-commit check (mandatory)**: refuses commits that mismatch expected encryption state, including the file-move trap.
9. **`bobusang reclassify` command**: the only supported way to change a file's encryption status.
10. **`bobusang verify-key-backup` command**: user-initiated key backup verification with timestamp recording.
11. **Session-start reminder hook**: nags on key-backup verification if stale.

## New failure modes v2.2 introduces (honest accounting)

1. **Key distribution is a user problem bobusang cannot solve.** Documented, but the cost of mishandling is borne entirely by the user.
2. **Key loss is terminal.** Structural defense (two-backup enforcement + nag) reduces probability but cannot eliminate.
3. **git-crypt tooling dependency** — install may fail on locked-down systems. `setup.sh` must abort cleanly with an actionable message, not silently fall through to plaintext.
4. **Mixed-encryption grep confusion** — `grep` across `memory/` won't see `secrets/` when locked. Documented.
5. **Merge conflicts on encrypted files** — git-crypt handles them in the unlocked working tree. If users resolve a conflict without all three sides decrypting cleanly (partial key state), they can write corrupted encrypted content. Must be tested; conflict-handling guidance in the docs.
6. **Binary-looking diffs on locked devices** — normal, documented.
7. **File-move silent re-encryption trap** — addressed by mandatory pre-commit check + `bobusang reclassify` command. Without the check, this would be a v2.1-class fatal flaw.
8. **`git archive` edge case** — if a user runs `git archive` on an unlocked working tree, the output is the **plaintext** working-tree contents (archive does not run filters). Documented: "do not use `git archive` to export memory repos unless you intend to ship plaintext."
9. **Fresh clone + user confusion** — on a device without the key, encrypted files appear as binary garbage in the working tree. If the user edits or resets them thinking the repo is corrupted, they can corrupt encrypted blobs (data loss, not secret leak). Documented in `multi-device-keys.md`.

## Platform matrix (gated)

bobusang claims multi-platform support. v2.2 may NOT claim support for any platform that has not been verified against git-crypt specifically. The matrix:

| Platform | v2.0 status | v2.2 status (before verification) | Verification task |
|----------|-------------|------------------------------------|-------------------|
| WSL2 Ubuntu | Fully supported | Assumed supported, must verify | Full setup→encrypt→commit→clone→unlock→edit→commit cycle |
| Linux (native) | Fully supported | Assumed supported, must verify | Same |
| macOS | "Should work" | **Not claimed** | Requires verification AND explicit test of APFS case-insensitivity with `.gitattributes` patterns (known failure mode) |
| Windows (native, Git Bash) | Partial | **Not claimed** | git-crypt has no official Windows binary. Requires MSYS2 or source build. Requires test of `core.autocrlf` interaction with encrypted files (known footgun). |
| Windows (WSL2) | Fully supported | Assumed supported, must verify | Same as Linux native |

**The README's platform table may not list a platform as supported for v2.2 until the verification task has been completed and recorded in `docs/design/platform-matrix-v2.2.md`.**

## Implementation spikes (must complete before locking design)

These are not open questions. They are unknowns that must be resolved by hands-on testing before implementation proceeds. Each produces a short memo committed alongside this design doc.

### Spike 1: `git pull` + smudge interaction on encrypted paths

Unknown: when a user on an unlocked device pulls new commits that touch encrypted files, does git-crypt re-smudge the working tree correctly without manual intervention? What happens when pulled content conflicts with local encrypted edits?

Must verify on throwaway repo. Failure mode = daily sync workflow broken.

### Spike 2: File-move detection in pre-commit check

Unknown: can the pre-commit hook reliably detect the `secrets/ ↔ domain/` file-move case using only `git diff --cached --name-status`? What about partial renames, copies, case-only changes on APFS? What about renames that also modify content?

Must verify. Failure mode = the file-move trap is unmitigated.

### Spike 3: Merge conflicts on encrypted files

Unknown: when two devices modify the same encrypted file and one pulls the other, what does the conflict state look like? Does git-crypt produce clean conflict markers in the decrypted working tree? Can a user resolve the conflict normally and re-commit?

Must verify. Failure mode = multi-device workflow breaks on normal conflict.

### Spike 4: Platform-specific gotchas

APFS case-insensitivity + `.gitattributes` case-sensitivity. `core.autocrlf` on Windows with encrypted binary-looking content. Both are known footgun categories. Need explicit test cases.

### Ecosystem audit (also blocking, described above)

Not a spike but a prerequisite. Listed here for completeness.

## Testing plan

- **Unit**: `setup.sh` paths for (a) git-crypt already installed, (b) not installed but installable, (c) not installable. Key export to expected location with correct permissions (0600). Two-backup prompt enforcement.
- **Integration**: full cycle on a throwaway repo — init, encrypt a file, commit, clone to "device B," unlock with exported key, verify plaintext, modify, commit, pull back to "device A," verify consistency.
- **File-move trap**: move a file from `secrets/` to `domain/` and verify pre-commit refuses the commit. Move via `bobusang reclassify` and verify it succeeds with the expected commit message.
- **Failure modes**: clone without key → `sync-memory.sh` errors clearly. Lose key simulation → docs match reality. git-crypt not installable → `setup.sh` aborts cleanly.
- **Platform matrix**: all spikes run on each claimed platform. No claim without a green run.

## Remaining genuine open questions

The six original open questions have been reclassified. Most were hiding spikes or design gaps:

1. ~~**Default encrypted paths**: only `secrets/` subdirs, or also encrypt `notes/{hostname}.md` by default?~~ **Genuine open question. Leaning: encrypt `secrets/` subdirs only by default, let users opt in to encrypting `notes/{hostname}.md` per device.**
2. ~~**Install strategy**: auto-install git-crypt via package manager, or print instructions?~~ **Genuine open question. Leaning: print instructions + provide a helper command. Auto-sudo is surprising.**
3. ~~**Key file default location**~~ — **resolved**: password manager is the storage, `~/.claude/bobusang-key.gpg` is only the transient export location during setup. Encourage deletion of the local file after backup is verified.
4. ~~**Pre-commit sanity check: opt-in, opt-out, or mandatory?**~~ — **resolved**: mandatory. Non-negotiable. Without it the file-move trap is unmitigated.
5. **Plaintext fallback UX**: when a user declines encryption, should bobusang remind them later? **Genuine open question. Leaning: silent. Respect the decision.**
6. ~~**`git pull` + smudge interaction**~~ — **reclassified as Spike 1. Must resolve before locking.**

## Rollout

- **Phase 0** (blocking): ecosystem audit + 4 spikes. No implementation.
- **v2.2.0**: implementation on platforms that passed the matrix. Interactive prompt in `setup.sh`. Default is still **no encryption** — existing users keep working. New users who say yes get the full path with mandatory safety checks.
- **v2.2.1**: fixes based on real-world install experiences.
- **v2.3**: if key-distribution pain is severe enough to drive users away, revisit with possibly an age-based alternative. Not committed.

## Dogfooding

`claude-config` (the author's private config repo, synced across 4 devices) is an ideal internal test subject for v2.2. It is genuinely multi-device, contains real sensitive context, and is run by the design author who can tolerate rough edges. The plan:

- v2.2 implementation → spike verification → internal dogfood on `claude-config` → 2+ weeks of real use → fixes → public release.
- Do NOT migrate `claude-config` before implementation and spike verification. That would skip the safety net.
- Migration of `claude-config` is tracked as a follow-up task, not a v2.2 deliverable itself.
