# v2.2 Design — Encrypted Memory

**Status**: Draft (rev 3 — post-audit, post-devil)
**Target**: bobusang v2.2
**Author**: @jun0-ds
**Supersedes**: [v2.1-private-tags.md](./v2.1-private-tags.md) (killed by devil)
**Revision history**:
- rev 1 (2026-04-14) — initial git-crypt design after v2.1 was killed
- rev 2 (2026-04-14) — 8 devil revisions on v2.2 itself (auto-unlock reality, password-manager-only key story, file-move trap, pre-commit mandatory, spike spin-off, structural key-loss defense, platform matrix gating)
- rev 3 (2026-04-15) — **primitive swap: git-crypt → transcrypt**. [Ecosystem audit](./ecosystem-audit-2026-04.md) found git-crypt's `master` dormant since 2018 with open merge breakage (#20, #273) unresolved. transcrypt is actively maintained and ships a custom merge driver that directly addresses the one failure mode bobusang cannot live with. Second devil round on the switch itself survived with required narrative downgrades, baked in here.

## Problem

Users of bobusang will write sensitive content into memory files: internal names, pricing, contact info, half-baked thoughts they don't want leaking. The git remote — even if private — is not a trust boundary that can hold forever. Private repos get breached, visibility flips by accident, hosting companies get compromised, old collaborators retain access. A multi-device memory system needs to assume the remote is eventually hostile.

v2.1 tried to solve this with `<private>` tags and a git clean filter. [Devil's advocate review](./v2.1-private-tags.md) found the design was fail-OPEN on fresh clones, destroyed secrets on routine `git checkout --`, and broke multi-device sync. It was killed.

v2.2 picks the honest primitive: **encrypt the files. Let the encrypted form travel through git. Manage keys out-of-band.**

A second devil round on v2.2 itself confirmed the architecture is sound (file-level encryption at the git boundary is the right layer) but surfaced weaknesses in how the design presented its own limits (rev 2). A third round — after the ecosystem audit flipped the primitive from git-crypt to transcrypt — survived with narrowed justification. Rev 3 bakes all of these into the main text.

## Threat model (explicit)

### What v2.2 defends against

1. **Remote repo leak, device not compromised.** GitHub visibility flips, backup gets exposed, account compromised via token leak. Attacker sees ciphertext blobs, not plaintext. This is the primary defended case.
2. **Long-lived access drift (narrow).** A collaborator from two years ago still has read access to the remote. They can't decrypt content committed after their key access was revoked — but only if the key was rotated, which the design treats as essentially not-happening in practice. In practice this defense is "access drift during the period before the collaborator ever had the key," which is narrower than the aspiration.
3. **Laptop theft with key stored separately.** If the decryption key is in a password manager or on a physical token and NOT cached on the stolen laptop, the thief has the repo but not the key.

### What v2.2 does NOT defend against (stated upfront, not buried)

1. **Compromise of a device with the key loaded.** If the attacker gets a live shell on an unlocked device, everything is readable.
2. **Auto-unlock convenience configurations.** The ergonomics of any encrypted-git tool push users toward caching the key on disk or in an agent with indefinite cache so `unlock` doesn't require a passphrase every session. **On any device configured this way, the "key is separate" assumption collapses.** Device compromise = key compromise. v2.2 does not prevent users from configuring auto-unlock, because the alternative (passphrase every session) is untenable for daily use. Realistically, most devices will end up auto-unlocked. **This makes v2.2 a one-factor defense against a remote-only leak, not a two-factor defense against arbitrary compromise.** Calling it anything else is dishonest.
3. **Commit messages, file names, directory structure.** The tool encrypts file *contents*. Names and structure are visible. Users must not put secrets in commit messages or filenames. No technical mitigation — documented as author responsibility and accepted as a leak vector.
4. **Git history from before v2.2.** Anything committed in plaintext before v2.2 is in history forever. `git filter-repo` is documented but not automated.
5. **Key compromise is retroactive and effectively unrecoverable.** transcrypt uses a symmetric repo password. If it leaks, *all* historical encrypted content is decryptable. With multi-month memory across 4 devices, users **will not rotate**. The honest recovery path is "stop committing new sensitive content to this repo and consider starting a new repo." Rotation is documented as a procedure, not presented as a routine mitigation.
6. **Compliance regimes.** Not HIPAA, not SOC2, not PCI. Use a real vault.
7. **Non-text secrets.** Images, screenshots, binary artifacts. transcrypt will encrypt them if globbed but diff/merge is poor on binaries; bobusang doesn't track them specifically.
8. **File-move silent re-encryption.** See "New failure modes" below. Moving a file between encrypted and plaintext directories silently changes its encryption status with no error and no user warning. Defended by a mandatory pre-commit check (see below), not by transcrypt itself.

### Summary of the honest story

v2.2 is **one-factor encryption at rest, primarily protecting against remote-only exposure**. It is not a vault, not a two-factor scheme, and not a protection against device compromise. It is a large improvement over "plaintext in a private repo that might someday leak," and that is the specific threat it targets.

## Why transcrypt (and not the alternatives)

The primitive choice comes from [`ecosystem-audit-2026-04.md`](./ecosystem-audit-2026-04.md). Short version:

| Option | Verdict | Notes |
|--------|---------|-------|
| **transcrypt** | ✅ **primary** | Actively maintained (last commit 2026-04-06, v2.3.1 in 2025-02). Bash + OpenSSL CLI. **Ships a custom merge driver added in response to #69 — exactly the bobusang failure mode.** Works with plain `git pull`/`merge`/`checkout`. No apt package in mainline Ubuntu (`setup.sh` must automate install). Windows native is messy; WSL2 works. |
| git-crypt | ⚠ fallback only | `master` dormant since **2018-02-16**. 0.8.0 (2025-09) was a side-branch OpenSSL compatibility hotfix, not active development. **Open merge-breakage issues (#20, #273) with git ≥ 2.39.1 — unresolved for multi-years.** One-command apt install is the only real advantage. Acceptable only if the team commits to "never edit encrypted files on two devices simultaneously" discipline — which earlier devil rounds proved is not trustworthy. |
| git-agecrypt | ⚠ v2.3 re-audit | Architecturally cleanest (age-based, identity files outside repo, no key material in repo). But no tagged releases, single maintainer, README explicitly disclaims security review. Too immature in 2026-04. Revisit when it grows up. |
| git-secret, sops, `<private>` tags, gitignored-per-device | ❌ | See v2.1 doc + audit memo for full rejection rationale. |

### The single load-bearing reason for the switch (per devil round 2)

The second devil's advocate round on "switch to transcrypt" found that most of the audit's pro-transcrypt narrative was over-sold: bus factor is still 1, "bash + openssl is a smaller trusted base" is an aesthetic claim dressed as security, Spike 3 is not actually pre-answered by a single closed issue. After those corrections, **one reason survives as solid**:

> **git-crypt's merge breakage (#273) is open, multi-year, unresolved, and directly conflicts with bobusang's daily operation (sync-memory.sh runs `git pull`/`rebase` hourly or faster). The only way to keep using git-crypt safely would be an author-discipline rule ("never edit encrypted files on two devices at once") — which is exactly the class of defense v2.1 devil already rejected as unreliable.**

Everything else ("actively maintained," "smaller trusted base," "merge driver fixes our problem") is **weaker evidence that needs spike verification** (see below). The switch is justified on the single fact that git-crypt is broken for our workflow, not on transcrypt being qualitatively better across the board.

### Honest limitations of transcrypt

- **Bus factor 1.** The audit softened this ("effectively 1 but with steady contributor PRs") but the underlying risk is the same as git-crypt's — one maintainer. The difference is "aware-and-active maintainer" vs "absent-since-2018 maintainer," which is a real but bounded improvement.
- **Bash + openssl subprocess surface.** Bash has its own footgun classes: quoting, word splitting, IFS, binary-in-pipe, shell injection if filenames contain metacharacters. Smaller trusted base ≠ safer trusted base. **Spike A must include a manual audit of transcrypt's clean/smudge/merge driver scripts for quoting and injection hygiene.**
- **apt install gap.** No mainline Ubuntu package. bobusang's `setup.sh` MUST fully automate the install (download release tarball, verify checksum, place in `~/.local/bin/`, chmod) or the UX degrades to "users have to run a manual install before setup.sh" which drops adoption.
- **Submodules unsupported** (issue #88 open). bobusang doesn't use submodules, so non-issue for us, but note it for future.
- **Windows native is painful.** bobusang officially supports WSL2 for Windows users. Native Windows support claim is dropped from the platform matrix.

### Fork option as last resort

If transcrypt goes dormant in 1-2 years, the ~1000-line Bash codebase is **small enough to fork and maintain internally** if absolutely necessary. This is not a primary mitigation — it's the existence-of-escape-hatch argument. git-crypt's 5000-line C++ would not be forkable by a solo maintainer. This is one of the few things that genuinely tips transcrypt's favor.

## Design

### What gets encrypted

`.gitattributes`:

```
memory/secrets/** filter=crypt diff=crypt merge=crypt
notes/secrets/**  filter=crypt diff=crypt merge=crypt
```

(`crypt` is transcrypt's registered filter name. The `merge=crypt` line is the one that makes merges work — not present in git-crypt's gitattributes, and that's the whole story.)

**Encryption is directory-scoped, not tag-scoped.** A file is either entirely encrypted or entirely plaintext, based on which directory it lives in. No regex parsing, no author discipline on a per-line basis.

```
memory/
├── core/              # plaintext — identity, work style
├── domain/            # plaintext — active projects (non-sensitive parts)
├── secrets/           # ENCRYPTED — anything sensitive
│   ├── credentials.md
│   └── project-x-secrets.md
└── archive/
    └── secrets/       # ENCRYPTED (same glob via **/secrets/)
```

Users who want to mix: split a file into `domain/project-x.md` (public) + `secrets/project-x.md` (private), and let Claude read both via a CLAUDE.md rule.

### The file-move trap (addressed, not ignored)

Directory-scoped encryption has a failure mode v2.1's tag-scoped approach did not: **moving a file between encrypted and plaintext directories silently changes its encryption status with no feedback.**

```bash
# User intent: "re-categorize this as public info"
mv memory/secrets/project-x.md memory/domain/project-x.md
git add memory/
git commit -m "move project-x to public section"
git push
# Result: entire file contents now plaintext on the remote
```

And the reverse:

```bash
mv memory/domain/foo.md memory/secrets/foo.md
# Result: future commits encrypted, but ALL prior plaintext history of foo.md
# remains in git history forever. User gets a false sense of "I encrypted it."
```

This is directly analogous to v2.1's "`git checkout --` destroys secrets" bug — a routine git operation silently changes encryption state without warning. v2.2 must not ship without a defense.

**Defense: mandatory pre-commit check.** A hook bobusang installs at `setup.sh` time. On every commit, it runs `transcrypt --list` (or equivalent content inspection) on all staged files and compares the expected encryption state (based on current `.gitattributes`) against the actual staged blob. If any staged file under `secrets/**` appears to be plaintext, or any file outside `secrets/**` that was previously encrypted is now plaintext (move detection via `git diff --cached --name-status`), the commit is refused:

```
bobusang: refusing commit — encryption state mismatch

  memory/domain/project-x.md appears to be plaintext but its prior
  revision at memory/secrets/project-x.md was encrypted.

  If you intend to publish this content in plaintext, run:
      bobusang reclassify memory/secrets/project-x.md --to-plaintext
  This will explicitly acknowledge the encryption change.

  If this was a mistake, move the file back to secrets/.
```

This check is **mandatory**, not opt-in. Without it, the file-move trap is unmitigated.

The `bobusang reclassify` command is the only supported way to change a file's encryption status. It decrypts (if needed), moves, optionally clears history if the user explicitly accepts the destructive rewrite, and re-commits with a recognizable commit message so the pre-commit check knows to allow it.

### CLAUDE.md integration

New section in the default `CLAUDE.md` template:

```markdown
## Secrets

Files under `~/.claude/memory/secrets/` and `~/.claude/notes/secrets/` are
encrypted at rest in git via transcrypt. On a device where the key is loaded,
they appear as normal markdown and should be read alongside their public
counterparts when relevant.

When you see `memory/domain/foo.md`, also check for `memory/secrets/foo.md`.
Treat the secrets file as the private appendix to the public one.

Never move files between `secrets/` and non-secrets directories with `mv` or
`git mv`. Use `bobusang reclassify` so the encryption state change is
explicit and recorded.
```

## Key lifecycle

### Key distribution: password manager is the only practical answer

Earlier drafts listed four options for getting the decryption key onto new devices (cloud with separate credentials, password manager, hardware token, physical transfer). In reality **only the password manager is practical for daily-use multi-device workflow**, for reasons spelled out in rev 2. The other three are acknowledged edge cases for users with specific constraints, not a flexible menu:

- **Cloud with separate credentials** requires genuine credential separation from the git remote. Most users recycle credentials, making the "separation" fiction.
- **Hardware token** falls back to "left plugged into one device" in practice.
- **Physical transfer (USB, printed QR)** is impossible for remote SSH-only servers and degenerates to SCP, which shares the SSH trust surface.
- **Password manager with file attachment** (1Password, Bitwarden, KeePassXC) — all support binary attachments, work on all platforms, practical for daily use. **This is the supported path.**

bobusang docs and `setup.sh` prompts recommend the password manager path as the first-class option and treat others as edge cases with warnings.

### Structural defense against key loss

A warning message ("⚠ if you lose this key, memory is unrecoverable") is not a defense. Users read and dismiss. The design enforces key backup at structural points:

1. **`setup.sh` requires two backup locations before completing encryption setup.**
   ```
   You are enabling encrypted memory. bobusang will not proceed until you have
   backed up the decryption key in two separate locations.

   Location 1 (e.g., your password manager): [prompt]
   Location 2 (e.g., encrypted USB, second password manager): [prompt]

   bobusang trusts you to have put the key there, but will remind you monthly to
   verify it. Bypass with --i-accept-single-point-of-failure.
   ```
2. **`bobusang verify-key-backup` command.** User runs it. bobusang prompts them to manually retrieve the key from each claimed backup location and confirm its SHA256 matches. On success, records the timestamp.
3. **Periodic reminder in session-start hook.** If `verify-key-backup` has not run in 60 days, session-start shows:
   ```
   bobusang: key backup last verified 73 days ago.
   Run `bobusang verify-key-backup` to confirm you can still recover your memory.
   ```
   Dismissible but nag-persistent. Nag is the point.
4. **No automatic cloud backup of the key.** Not even opt-in. The moment such a feature exists, it becomes a target and a crutch. Users who want belt-and-braces use their existing backup tooling on the key file.

### Initial setup (device A)

```
$ bash setup.sh
...
Enable encrypted memory (via transcrypt)? [y/N] y

Checking audit memo: docs/design/ecosystem-audit-2026-04.md → transcrypt confirmed.

Installing transcrypt...
  [detects platform; on Ubuntu WSL2: download release tarball,
   verify checksum, chmod, place in ~/.local/bin/; on macOS:
   `brew install transcrypt`; on other: print install instructions
   and abort cleanly]

Initializing transcrypt in this repo...
Exporting key via `transcrypt --display`.

⚠ This password IS the decryption key for all your encrypted memory.
⚠ Back it up in TWO separate locations before proceeding.

Location 1 (e.g., password manager): [prompt]
Location 2 (e.g., encrypted USB):    [prompt]

Installing pre-commit check... ok
Installing session-start key-backup reminder... ok

Writing .gitattributes with:
  memory/secrets/** filter=crypt diff=crypt merge=crypt
  notes/secrets/**  filter=crypt diff=crypt merge=crypt

Done. See docs/guides/multi-device-keys.md for adding other devices.
```

### Adding a new device (device B)

Documented in `docs/guides/multi-device-keys.md`. The procedure:

1. Retrieve the key from the password manager on device B.
2. `git clone <remote> && cd <repo>`
3. `transcrypt -c aes-256-cbc -p <key>` (or interactive prompt)
4. `bash setup.sh` — bobusang recognizes the transcrypt state and installs the pre-commit check and session-start hooks without re-initializing.
5. Optionally run `bobusang verify-key-backup` to confirm device B's retrieval path still works.

### Key rotation (honest answer)

**Realistic answer: users will not rotate.** Rotating transcrypt's key requires decrypting everything, re-initializing, re-encrypting, force-pushing, and coordinating all devices to re-unlock with the new key. With 4 devices and months of memory, this is a migration project, not a routine.

The documented response to key compromise is therefore **not rotation**. It is:

1. Stop using the current repo for any new sensitive content immediately.
2. Treat the current repo's encrypted content as compromised going forward.
3. Decide whether to migrate to a fresh repo with a new key (accepting history-continuity loss) or accept the compromise.

Not glamorous, but honest. Pretending rotation is a routine recovery path invites users to underinvest in protecting the current key.

## What bobusang actually builds

1. **Audit memo** ([`ecosystem-audit-2026-04.md`](./ecosystem-audit-2026-04.md)) — ✅ done (unblocks implementation)
2. **`setup.sh` integration**: transcrypt detection, installation automation (download release, verify checksum, PATH placement), init, key export, **two-backup enforcement**, pre-commit hook install, session-start reminder install
3. **`.gitattributes` template** with `memory/secrets/**` and `notes/secrets/**` patterns (including `merge=crypt`)
4. **`CLAUDE.md` template update** with the Secrets section
5. **`docs/guides/multi-device-keys.md`** — long-form honest guide. Password manager as first-class path.
6. **`sync-memory.sh` awareness**: detect transcrypt lock state and refuse to sync with clear message if key isn't loaded. Don't silently sync garbage.
7. **`detect-changes.sh` awareness**: don't falsely flag "all files changed" because key isn't loaded.
8. **Pre-commit check (mandatory)**: refuses commits that mismatch expected encryption state, including the file-move trap.
9. **`bobusang reclassify` command**: the only supported way to change a file's encryption status.
10. **`bobusang verify-key-backup` command**: user-initiated key backup verification with timestamp recording.
11. **Session-start reminder hook**: nags on key-backup verification if stale.
12. **Merge-conflict runbook**: `docs/guides/merge-conflicts.md`. Even with transcrypt's merge driver, three-way conflicts on encrypted files need human resolution and specific steps to avoid re-encrypting garbage.

## New failure modes v2.2 introduces (honest accounting)

1. **Key distribution is a user problem** bobusang cannot solve beyond documentation.
2. **Key loss is terminal.** Two-backup enforcement + nag reduces probability but cannot eliminate.
3. **transcrypt install** may fail on locked-down systems. `setup.sh` must abort cleanly, not silently fall through to plaintext.
4. **Mixed-encryption grep confusion** — `grep` across `memory/` won't see `secrets/` when locked. Documented.
5. **Merge conflicts on encrypted files** — transcrypt handles them via merge driver, but three-way conflicts still need human resolution. **Users who resolve conflicts carelessly can write corrupted content back.** The runbook (#12 above) is a v2.2 deliverable, not a post-incident cleanup.
6. **Binary-looking diffs on locked devices** — normal, documented.
7. **File-move silent re-encryption** — addressed by mandatory pre-commit check + `bobusang reclassify`.
8. **`git archive` edge case** — filters don't run on `git archive`. If a user runs it on an unlocked working tree, output is plaintext. Documented: "do not use `git archive` to export memory repos."
9. **Bash script injection surface** — filename and content going through shell pipes. Spike A must manually audit transcrypt's clean/smudge/merge scripts for quoting hygiene.

## Implementation spikes (BLOCKING)

These are not open questions. They are unknowns that must be resolved by hands-on testing before implementation proceeds. Each produces a short memo committed alongside this design.

### Spike A — transcrypt full-stack verification

1. **Fresh install on WSL2 Ubuntu** — document the actual path. If it requires manual steps beyond `setup.sh` automation, that's a rev 4 finding.
2. **Full cycle**: init, encrypt a file, commit, clone to "device B," unlock with the exported key, verify plaintext, modify, commit, pull back to "device A," verify consistency.
3. **Manual audit of transcrypt's bash scripts** (clean/smudge/merge driver) for quoting and injection hygiene. Specifically: what happens when a filename contains spaces, newlines, `$`, backticks, or shell metacharacters. What happens when file content contains null bytes. What happens when `git merge` runs the driver with unusual argument patterns.
4. **Verify the claim that v2.3.1 is quality, not churn** — read the last 10 commits, confirm they're fixes or maintenance, not experimental rewrites.

### Spike B — git-crypt merge breakage reproduction (disconfirmation test)

Run against the current system git version (2.43+ on modern Ubuntu):

1. Reproduce issue #273 on a throwaway repo with git-crypt installed
2. If reproducible → transcrypt switch confirmed, proceed to Spike C
3. If NOT reproducible → revisit whether git-crypt is actually broken for our workflow, rev 4 may revert the primitive choice

### Spike C — pre-commit check reliability

1. Implement a minimal pre-commit hook using `git diff --cached --name-status`
2. Test file-move detection: `secrets/ → domain/`, `domain/ → secrets/`, rename-with-content-change, case-only rename on case-insensitive filesystems
3. Test false positives: re-staging an already-encrypted file, re-adding an already-plaintext file
4. Confirm the hook blocks the bad cases and allows the good cases

### Spike D — merge conflict edge cases with transcrypt's driver

1. Two devices modify the same encrypted file concurrently → pull → conflict state. Does transcrypt's merge driver produce clean plaintext conflict markers? Can a user resolve normally and re-commit?
2. One device renames `secrets/foo.md → secrets/bar.md`; other device edits `secrets/foo.md` → merge conflict type: rename+modify. Does the driver handle it?
3. `.gitattributes` itself conflicts → what's the state of encryption rules during resolution?
4. `stash pop` after a pull with encrypted conflicts

Each spike produces a `docs/design/spike-X-<name>.md` memo with the test harness, findings, and go/no-go for rev 4.

### Spike failure policy

If **Spike A or Spike D fails** → rev 4 required, possibly reverting to git-crypt-with-discipline or pausing v2.2 entirely.
If **Spike B shows issue #273 is fixed** → rev 4 may revert primitive choice; dormant master is then only a bus-factor concern.
If **Spike C fails** → pre-commit check needs a different implementation strategy (could be a repo-state comparison rather than diff-based).

## Open questions (genuine — remaining after reclassification)

1. **Default encrypted paths**: only `secrets/` subdirs, or also encrypt `notes/{hostname}.md` by default? **Leaning: secrets/ only by default, users opt in per-device if they want inbox encrypted.**
2. **Plaintext fallback UX**: when a user declines encryption, should bobusang remind them later? **Leaning: silent. Respect the decision.**

All other rev 2 open questions are now either resolved (key location → password manager only, pre-commit → mandatory, ecosystem → transcrypt) or reclassified as spikes.

## Rollout

- **Phase 0** (blocking): ecosystem audit (✅ done) + 4 spikes. No implementation until all green.
- **v2.2.0**: implementation on platforms that passed the matrix. Interactive prompt in `setup.sh`. Default is **no encryption** — existing users keep working. New users who say yes get the full path with mandatory safety checks.
- **v2.2.1**: fixes based on real-world install experiences.
- **v2.3**: re-audit. git-agecrypt may have matured. transcrypt maintenance status re-checked. If either has changed materially, revisit primitive choice.

## Dogfooding: claude-config

The author's private config repo (`jun0-ds/claude-config`) is an ideal internal test subject. It is genuinely multi-device (4 devices), contains real sensitive context, and is run by the design author who can tolerate rough edges. The plan:

- v2.2 implementation → spike verification (all green) → internal dogfood on `claude-config` → 2+ weeks of real use → fixes → public release
- **Do NOT migrate `claude-config` before spike verification.** That would skip the safety net.
- **Phase A pre-stage is already done** (as of 2026-04-14): `claude-config` has `memory/secrets/` and `memory/archive/secrets/` populated, `**/secrets/` gitignored, CLAUDE.md §8 writing rules in place. The migration is reduced to "change `.gitattributes` to encrypt these existing paths" once v2.2 ships. See `claude-config`'s `d7f5870` commit.
- Historical plaintext cleanup (voyage_migration, milvus_ncp_migration production IPs) is a separate decision from v2.2 migration — likely answer is "treat leaked IPs as leaked, rotate at network layer, do not rewrite git history."
