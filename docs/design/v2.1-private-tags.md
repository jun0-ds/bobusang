## ⚠ Why this was killed

Devil's advocate review on 2026-04-14 flipped or weakened nearly every load-bearing claim.
Kept here as a historical record so the same mistake isn't made twice.

**Fatal flaws found:**

1. **Fail-OPEN, not fail-closed.** `required = true` in `.git/config` only catches runtime filter errors. `.git/config` is not committed — every fresh clone has no filter registered. `gitattributes(5)`: *"If the filter command is not defined in the configuration, the attribute is treated as if it were not set."* Result: a user on a fresh device writes `<private>secret</private>`, `git add` silently passes it through, secret lands on the remote in plaintext. The mechanism is fail-OPEN by default.

2. **Multi-device sync breaks the core value prop.** `smudge = cat` means device B only ever sees `[REDACTED]` after `git pull`. The secret stays on device A and nowhere else — effectively per-device. bobusang exists to sync memory across devices; a secrets mechanism that doesn't sync is a contradiction.

3. **`git checkout --` silently destroys secrets.** Any smudge-triggering operation (`checkout --`, `reset --hard`, `stash pop`, branch switch, pull) overwrites the working tree from the index. Index has `[REDACTED]`. Working tree loses the original. No recovery. Everyday git operations become destructive.

4. **Opt-in contradicts fail-closed.** A security feature users can skip is a security feature users will skip. Opt-in + fail-closed cannot both be true.

5. **Author-controlled = one typo leaks.** `<privat>`, a forgotten tag, a secret in a commit message (filters don't touch commit messages), a secret in a markdown code fence the regex doesn't understand — all leak. Security that depends on author discipline isn't security.

6. **History never gets scrubbed.** `--renormalize` punted to the user means pre-v2.1 secrets remain in git history forever. The most important case — "clean up what's already leaked" — is the one v2.1 doesn't touch.

7. **Wrong abstraction borrowed from claude-mem.** claude-mem strips inside its own daemon pipeline, *before* anything reaches git or disk. bobusang copied the surface syntax (`<private>`) but tried to apply it at the git boundary — a completely different architectural layer. The syntax made the design look familiar; the location made it fail.

**False confidence trap**: the first successful `<private>` test creates a mental model of "bobusang protects me," which causes the user to stop double-checking and eventually leak a secret they forgot to tag. Security theater that actively increases risk.

**Verdict**: v2.1 had the complexity of a pre-commit hook, the security of a gitignored file, and the false confidence of a vault. Worst of all three.

## Problem

bobusang syncs memory files to a git remote. Users inevitably accumulate sensitive fragments inside those files — API keys, internal names, contact info, pricing, credentials — because memory is meant to capture *working context*, and real working context includes secrets.

Right now bobusang offers no answer for this. Users either:

1. Keep the git remote private (works, but forecloses sharing, open-sourcing configs, or using bobusang on public infra)
2. Manually redact before commit (error-prone, breaks the "git = source of truth" flow)
3. Don't write sensitive details into memory at all (defeats the point of memory)

None of these scale. bobusang needs a first-class mechanism so that **memory files can live locally with full detail, and the git remote only ever sees a safe projection**.

## Non-goals

- **Not** a secrets manager. bobusang won't store, rotate, or inject secrets into running processes.
- **Not** encryption. v2.1 strips; it doesn't encrypt. (Encryption is v2.2+ via git-crypt guide.)
- **Not** a compliance tool. If you need HIPAA/SOC2, use a proper vault.
- **Not** retroactive. v2.1 handles new writes; cleaning historical commits is the user's problem (BFG, `git filter-repo`).

## Design principles

1. **Git stays clean.** No dirty working tree, no skip-worktree hacks, no "remember not to commit that file". The working tree and what git sees must be mechanically reconciled.
2. **Claude sees everything locally.** The whole point of memory is that Claude reads it. Stripping must happen at the git boundary, not the filesystem boundary.
3. **Author-controlled.** The user marks sensitive regions explicitly. No heuristic redaction — too many false positives and negatives.
4. **Fail closed.** If the strip filter is missing or broken, the commit should fail loudly, not silently leak.
5. **Reversible opt-in.** Users who don't care about this (private remote, solo use) should be able to skip it entirely with zero friction.

## Mechanism: git clean filter

Git's `clean`/`smudge` filter pair is the standard way to transform files at the index boundary:

- **clean** runs when content goes from working tree → index (on `git add`)
- **smudge** runs when content goes from index → working tree (on `git checkout`)

bobusang v2.1 registers a `clean` filter that strips `<private>...</private>` blocks. The `smudge` side is a no-op (`cat`) because the working tree is already the full version.

```
Working tree (full)    ──clean──▶   Index/remote (stripped)
        ▲                                    │
        └────────────── smudge (cat) ────────┘
```

### `.gitattributes`

```
memory/**/*.md filter=bobusang-private
notes/**/*.md  filter=bobusang-private
```

Scope: memory and notes only. `CLAUDE.md`, `settings.json`, `hooks/` are config and shouldn't contain `<private>` regions — if they do, that's a different problem.

### Filter registration

`setup.sh` registers the filter in the repo's local `.git/config`:

```
[filter "bobusang-private"]
  clean = bobusang-strip-private
  smudge = cat
  required = true
```

`required = true` is load-bearing: if `bobusang-strip-private` is missing from PATH, git fails the operation instead of silently passing content through. Fail closed.

### The strip script

`bin/bobusang-strip-private` — reads stdin, writes stdout, drops `<private>...</private>` blocks.

```bash
#!/usr/bin/env bash
# bobusang-strip-private: remove <private>...</private> regions from stdin
# Used as a git clean filter. Fails closed on malformed input.
set -euo pipefail

# Multiline, non-greedy strip. Replaces the block with a marker so diffs remain legible.
perl -0777 -pe 's{<private>.*?</private>}{<private>[REDACTED]</private>}gs'
```

Why perl and not sed/awk:
- `-0777` slurps the whole file (sed is line-oriented, awful for multiline)
- Non-greedy `.*?` with `s` flag handles nested markdown cleanly
- perl is installed everywhere bobusang targets (WSL2, Linux, macOS)

Why leave a `[REDACTED]` marker instead of deleting the block:
- Diff readability: removing content silently makes reviews harder
- Signals intent: someone reading the git version sees that *something* was scrubbed, not that the doc is incomplete
- Protects against accidental double-commit: if the clean filter runs on already-clean content, it's idempotent

### Tag syntax

```markdown
Public context about the deployment.

<private>
Prod DB host: db-internal.example.com:5432
Emergency contact: alice@company.internal
API key hint: rotates monthly, see 1Password "prod-api"
</private>

More public context.
```

Rules:
- Opening and closing tags on their own or inline — both work
- Nesting is undefined behavior (don't)
- Tags must match; a dangling `<private>` causes the strip to consume to EOF, which is wrong — the filter should detect and refuse (see "fail closed" below)

### Fail-closed validation

The strip script MUST refuse to pass content through if:

1. `<private>` count ≠ `</private>` count (dangling tag)
2. Nested `<private>` inside another `<private>` (ambiguous)

In either case: print error to stderr, exit non-zero. Git clean filter failure blocks the commit. The user sees a clear error and fixes the source file.

## Setup flow

`setup.sh` additions:

1. Detect if `bin/bobusang-strip-private` is in PATH; if not, install it to `~/.local/bin/` (or `~/.claude/bin/`).
2. Register the clean filter in the repo's `.git/config` (not global — this is repo-scoped).
3. Create `.gitattributes` if not present, or append the filter line.
4. Run `git add --renormalize memory/ notes/` once to retroactively apply the filter to existing tracked files. **Warn the user loudly** that this rewrites the working index and they should review the diff before committing.
5. Print a one-screen explainer of how to use `<private>` tags.

## Opt-out

Users with private remotes may not want this overhead. `setup.sh` should ask:

```
Does this repo sync to a public remote, or does it contain sensitive memory?
  [y] Yes — enable <private> tag stripping (recommended)
  [n] No — skip (my remote is private and trusted)
```

Opting out: no filter registered, no `.gitattributes` entry. Users can enable later via `bobusang enable-private-tags` (a new command in v2.1).

## What this does NOT solve

- **History leaks.** If a secret was committed before v2.1, it's in history forever. Document `git filter-repo` in the guide; don't try to automate it.
- **Accidental commits outside tags.** If the user writes a secret in plain text without wrapping it, the filter does nothing. This is author-controlled by design.
- **Non-text secrets.** Binary files, screenshots containing credentials, etc. Out of scope.
- **Sharing the unredacted version across devices.** v2.1 assumes the working tree *is* the full version on every device. If you want multi-device access to the same secrets, you need a sync path outside git (v2.2 will document git-crypt for this).

## Open questions

1. **Marker text**: `[REDACTED]` vs `[private]` vs empty. Leaning `[REDACTED]` for unambiguity.
2. **Tag name**: `<private>` vs `<secret>` vs `<redact>`. `<private>` reads most naturally in prose ("this is private"), and matches the claude-mem convention that inspired this.
3. **Scope `.gitattributes`**: only `memory/` and `notes/`, or everything? Starting narrow; can widen later.
4. **Install location**: `~/.local/bin/` (XDG-friendly) vs `~/.claude/bin/` (bobusang-owned). Probably `~/.local/bin/` — fewer surprises, standard PATH.
5. **Should `setup.sh` auto-enable, or ask?** Leaning ask — this is a meaningful behavior change and deserves explicit consent.

## Testing plan

- Unit: strip script handles balanced tags, dangling tags, nested tags, empty blocks, tags across multiple blocks in one file, unicode content.
- Integration: set up a throwaway repo, add a file with tags, `git add`, verify index content is stripped and working tree is intact.
- Regression: `git checkout` a stripped version, verify smudge is no-op and working tree stays stripped (important caveat — smudge cannot recover the original).
- Failure modes: remove `bobusang-strip-private` from PATH, verify `git add` fails with a useful error (thanks to `required = true`).

## Rollout

- v2.1.0: feature lands behind opt-in prompt in `setup.sh`.
- v2.1.1: bug fixes based on real-world usage.
- v2.2: companion guide for git-crypt (for users who need the unredacted version synced).
